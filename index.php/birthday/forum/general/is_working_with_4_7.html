Hello Anselme Chorein,

Thank you for posting to this thread your suggested solution.

I had avoided addressing the topic of the split / forks of the birthday datatype solution for a long time waiting for more progress to be made publicly before speaking on this topic. Unfortunately I kept putting this post off for far too long.

What many users don't realize is that long ago (some time after 4.3+ api changes) the svn repository for this project became unmaintained, which for end users / developers is very unfortunate.

Fortunately from this negative sprung a different kind of positive result ... forks (from what I have seen a fair number of them). First private forks (with only required or minimal modifications) then later public and unique forks.

Most of the forks are fairly similar, I think because this solution was created and refined early on in it's lifetime / history and as they say, it it ain't broke ... no need to fix it :)

Today I quickly see these forks:

- Xrow: While have not seen a public xrow git repository of this extension. I have seen that they have a version of the datatype with the xrowforum extension (though I do wish they would maintain a submodule instead). Though it seems that little to no continued improvements have been made to this copy, https://github.com/xrowgmbh/xrowf...atypes/ezbirthday/ezbirthdaytype.php

- eab-dev: This fork seems to be a full history import of the svn repository, patched with the required 4.3 api changes (ezpI18n::tr) and a helpful though slightly inaccurate README.md file, https://github.com/eab-dev/birthday/

---- Users who might consider using the svn repository provided by this project would be better served in the long run to use this git repository instead.

- Netgen: This fork seems to be import of the svn sources (without it's history), patched with the required 4.3 api changes (ezpI18n::tr) but without a helpful accurate README.md file, https://github.com/netgen/birthday

---- I think that users who use composer to install extensions should consider using the Netgen fork over the other sources. Also Netgen does very good eZ Publish work in the community and has proven themselves open to collaboration on public extension Pull Requests for bugfixes and other helpful improvements.

In light of all that I have shared, the next time I want to depend on a vendor's fork of this extension datatype (with respect to the vendors history of proper maintenance of a public resource and collaboration on it's development) ... I would choose to build upon the Netgen fork first.

In short: Please do not use the (eZ Publish <= 4.2) svn repository copy of this extension any longer unless your using a copy of eZ Publish 4.2 (or lesser version). Everyone else should use one of the publicly available Git based repositories instead (which have already been updated properly to support versions of eZ Publish 4.3 (or greater version))

If I have missed or omitted another public fork of this extension datatype, which is more than just a git repo with the needed 4.3+ api bugfixes / prior svn version history ... please share a link here in this forum thread so that others can properly consider which fork will best suite their needs and requirements.

I hope this summary helps others!

Take it eZ!
