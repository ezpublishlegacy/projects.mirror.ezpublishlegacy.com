CustomEdit

Last updated: Tuesday 01 September 2009 19:17

UNIX name Owner Status Version Compatible with
customedit *- pike planning 0.8 4.0.3

This extension allows a developer to write templates to
create, draft, edit, hide, move and delete nodes and versions without using

/content/action and /content/edit calls. It allows/requires
you to write your own, customized, edit forms that use a well
documented and easy to understand interface.
It's not intended as a replacement for the admin, but rather
as a simplified framework to build custom edit interfaces for the
public
user or intermediate parties (editors).
Please note
* This is not the default way to create forms in an ezPublish site.
Documentation on the default way of doing this is in the book, page
149-152. I can't find it online.
* This is in Beta. Things will change. Help is appreciated.

It has a number of advantages and limitations compared to the
/content/action and /content/edit framework of ezpublish:
* form field names are classattribute identifiers, not attribute
identifiers,
so you simply write
* as a result, you can create a node and enter its fields in one
form:
* as another result, you don't have to (and can not) use the
attribute_edit_gui widgets. this allows you to use different widgets

in several places, like, use tinymce on a textfield in one place, and

a plain textarea in another.
* as a disadvantage, the datatype limitations are not supported out

of the box. they are not implemented in ezPublish at a sufficiently
deep
level.
* as a disadvantage, you can't edit multiple nodes at the same
time
* you don't go through a whole range of redirects. you are in control
of
which templates are used for errorpage and successpage.
* as a result, you can pass variables between these templates.
* a new draft is only created at your request.
* as an advantage, you can edit published versions as they are,
withour creating new drafts, if you have the right to do so.
* the interface is a simplification of the edit process. You are
assumed to create a node on a location and edit it, or create
new versions, from there. The notion of objects is avoided at all

times in the http and php interfaces.
* as a disadvantage, a new node is always published. I see
no way to publish a node with only a draft object in the
php interfaces. Any help is appreciated.
All functionality is written on top of existing ezPublish
functioncalls. Triggers etc. are supposed to be performed as
usual.
This module extension uses the ContentObjectAttribute's
fromString() and toSring() methods, available since ezp 3.8

USAGE
There are several ways to use this extension:
After running the testsuite, you can
* write your templates that use the 3 main views as they are.
* write additional views, based on the first 3,
that have additional custom features.
* write completely different views, that use the nice
simplistic static methods of CustomEditLib.php
In all cases, I'm curious what you do and how you
get along. I'm sure there are bogz. Let me know !
