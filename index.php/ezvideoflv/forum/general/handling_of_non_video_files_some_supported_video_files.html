Hey Damien

I've been doing some testing with various file formats and doing a bit of digging though the code. I have made the following observations:

1) You can upload non movie files.
The templates display the "FLV version not yet generated" message which is misleading. As a user I'd expect to be told I was stoopid :)

What are your thoughts on checking the uploaded file format in eZVideoFLVType::validateObjectAttributeHTTPInput and if it is not a "supported" format failing the validation and displaying an appropriate error message?

2) Some movie files not supported although they are supported by ffmpeg-php.
Had do do a bit of digging for this one. eZVideoFLV::doConvert checks the MIME info using eZMimeType::findByFileContents and if it's not reported as belonging to the video group the conversion is abandoned (Silently returns null - it would be good if there was an error generated here)

The trouble is that eZMimeType::findByFileContents doesn't actually do what the name implies (working out the mime info via the file contents). It just passes off the call to eZMimeType::findByURL which utilises an hash of file suffixes/ MIME types (defined at the end of lib/ezutils/classes/ezmimetype.php )

The video file suffixes supported by eZMimeType are:



       array( 'mpeg', 'video/mpeg' ),


       array( 'mpg', 'video/mpeg' ),


       array( 'mpe', 'video/mpeg' ),


       array( 'mov', 'video/quicktime' ),


       array( 'qt', 'video/quicktime' ),


       array( 'avi', 'video/x-msvideo' ),


       array( 'movie', 'video/x-sgi-movie' ),


       array( 'rv', 'video/vnd.rn-realvideo' ),


       array( 'flv', 'video/x-flv' ),


       array( 'asf', 'video/x-ms-asf' ),


       array( 'wmv', 'video/x-ms-wmv' )


Files with extensions other than those listed are returned with the default MIME info - application/octet-stream. The list covers most of the popular ones but knowing users they'll try anything :)

There is a note in eZMimeType that says 
Should be removed to a .ini file or other external file
, so it has been noted. At least if this was the case the extension could define more mime types.

Not a lot that can be done with this one except maybe blindly trying to convert any files with an application/octet-stream MIME type and relying on return code from the conversion program to test for failure.

I think this could be mitigated by returning a "Uploaded file was not a supported file format" type error message with the file is uploaded. See 1).

P.S. Happy to contribute code by supplying patches or working on the repository - let me know what works for you.
