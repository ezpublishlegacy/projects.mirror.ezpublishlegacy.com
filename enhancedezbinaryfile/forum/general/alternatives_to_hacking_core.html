The short answer is that if it's supposed to be part of the collected information process, you will not be able to upload a file without a kernel hack.

The first question is - do you really want or need whatever you are doing to be part of collected information?

It's easy enough to create a form that submits to a custom module that creates a node with any uploaded files as a binary file object and send an email with the attributes of that node including a file (I've done this a few times).

The question is, do you really want users doing that? If it's a forum and people have to log in to upload files then it's probably fine - but, then, the information collection system isn't really suited for that. If you just want some way that even anonymous users can upload files (temporarily) that shouldn't be in the database - that's what this extension is for.

The original use for this extension was for people to upload resumes - which is something you probably don't want accessible in the database but do want to be able to route to someone via email.

I'm not sure what the question is "for accessing the results of a submitted file" - depending on your settings, the link to the uploaded file in the var directory should be in the admin template and the mail. That file will be available over http. If the link isn't showing, then something else is wrong (is the file there? it won't be without the kernel hack) or check your settings - the display of the link is determined by the settings.

Actually, the kernel hack is only to add the file to the notification email - so you can can get away with not using that in your case...

The problem though is that the information for the file is stored in the ezinfocollection_attribute table as xml - with an entry to the location of the file in the var directory (something like this: <siteaccess>/storage/original/attachments/application/5a5ae635ec635f65dccbb0b184c11944.pdf) and the original filename not in the binary file table. The content download link is what returns the hash file from the direct hash link with the current filename.

It would actually make more sense (to me) to use the existing file upload functionality (which can be seen with multifileupload and the plupload extensions) instead of the collected information - is there a reason why you are using the collected information functionality?

The more I think about this the more I think it "could" work with some modifications maybe, but it's not really the best way to go.

Here is a snippet of code that I used to create a file - the part that does it is the $dataType->insertRegularFile - this might help you narrow down what you want to look for:




if (count($errors) == 0 AND $_FILES['file']['size'] > 0 ) {


       if (!file_exists($targetDir))


               @mkdir($targetDir,0775, true);


       // Clean the fileName for security reasons


       $fileName = preg_replace('/[^\w\._]+/', '', $_FILES['file']['name']);


       // Read binary input stream and append it to temp file


       $content = file_get_contents($_FILES['file']['tmp_name']);


       file_put_contents( $targetDir . DIRECTORY_SEPARATOR . $fileName, $content);


       $upload = $dataType->insertRegularFile( $editObject, $version, $language, $attribute,$targetDir . DIRECTORY_SEPARATOR . $fileName, &$result );


       @unlink($targetDir . DIRECTORY_SEPARATOR . $fileName);


 

       if ($upload != TRUE) {


               $errors[] = ezpI18n::tr( 'design/gbw', "Upload failed");


       }


       if($result['errors']) {


               $errors = array_merge($errors,$result['errors']);


       }


}

"external visitors, people without an account or any access to site administration" - that's the Anonymous user - and if you want comments to be created by Anonymous users you can set Anonymous->content create->class comment in the roles - and then you should be able to have anonymous users create a comment. If you the add a file attribute to the comment class and add that to your comment edit templates then they should be able to upload a file using the standard "edit" functionality and not the "collection" functionality.

I think that's what you want. Or, at the very least that's the ezpublish way of doing comments. It doesn't make sense to use the information collection process since you are already creating a "comment" object (with the edit functionality) so you won't have an information collection object.

There are still - perhaps - reasons to use the information collection process... but if you can come up with a reason I can probably come up with the better way for the same functionality without using the information collection process. Of course if you are bound to using the information collection process for whatever political bs internal reason then this extension can probably be easily modified to easily ge the file path information.

Hmm... I'm not sure I'm understanding this 100%:

you want an object created with at least two attributes - a comment and a link to a file object (I assume a text line for a name too).

Anonymous users should not be able to read the comment but they should be able to get the link to download the file.

Admin user(s) should be notified when someone creates this object.

I'm assuming the file should be permanent with a link in the ezbinaryfile table.

You have access to the roles.

You do not have access to the kernel.

If the above is all true then I think the edit functionality is probably better to use since the file is then its own object and is in the ezbinary file. If the file should be temporary then the collection functionality using this extension is better (but not the kernel hack).

The comment attribute can easily be hidden with an if "is_logged_in" statement around the attribute in the front-end view template for the object class.

The notification can probably be done using the notification system (assuming the crons are set up etc.)

If it can't be done with the notifications because the crons won't be set up... or there is something else that won't work because of the setup there, then you can always go with a completely custom extension - which could create an object, set it to hidden or set it to a section that can be controlled with roles, and upload the file, set it in the ezbinaryfile table and send emails to whomever. It's not a particularly hard extension to do.
